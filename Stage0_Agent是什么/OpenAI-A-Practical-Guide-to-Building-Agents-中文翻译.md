## 引言

大语言模型在处理复杂、多步骤任务方面的能力日益增强。推理、多模态和工具使用方面的进步，解锁了一种全新类别的 LLM 驱动系统——**智能体（Agents）**。

本指南专为正在探索如何构建首个智能体的产品和工程团队设计，提炼了来自众多客户部署的洞察，形成实用且可操作的最佳实践。它包含了识别有前景用例的框架、设计智能体逻辑和编排的清晰模式，以及确保智能体安全、可预测和有效运行的最佳实践。

阅读本指南后，你将掌握自信地开始构建第一个智能体所需的基础知识。

## 1. 什么是智能体？

传统软件让用户能够简化和自动化工作流，而智能体能够**以用户的身份**，以**高度的独立性**执行同样的工作流。

> **智能体是能够独立代表你完成任务的系统。**

工作流（Workflow）是为实现用户目标而必须执行的一系列步骤，无论是解决客户服务问题、预订餐厅、提交代码变更，还是生成报告。

集成了 LLM 但**不**使用 LLM 来控制工作流执行的应用——比如简单的聊天机器人、单轮 LLM 调用或情感分类器——**不是智能体**。

更具体地说，一个智能体具备使其能够可靠且一致地代表用户行动的核心特征：

1. **它利用 LLM 来管理工作流执行并做出决策。** 它能识别工作流何时完成，并能在需要时主动纠正自己的行动。在失败的情况下，它可以停止执行并将控制权交还给用户。
2. **它可以访问各种工具以与外部系统交互**——既可以收集上下文，也可以采取行动——并根据工作流的当前状态动态选择适当的工具，始终在明确定义的护栏（Guardrails）内运行。

## 2. 何时构建智能体？

构建智能体需要重新思考你的系统如何做决策和处理复杂性。与传统自动化不同，智能体特别适合那些传统确定性方法和基于规则的方法难以应对的工作流。

以支付欺诈分析为例。传统的规则引擎像一份检查清单，根据预设标准标记交易。相比之下，LLM 智能体更像一位经验丰富的调查员——评估上下文、考虑微妙的模式、识别可疑活动，即使没有违反明确的规则。这种细致的推理能力正是智能体能够有效管理复杂、模糊情境的原因。

在评估智能体可以在哪里增加价值时，优先考虑那些之前难以自动化的工作流，尤其是传统方法遇到瓶颈的地方：

| 场景 | 说明 | 示例 |
|------|------|------|
| **① 复杂决策** | 涉及细微判断、异常情况或上下文敏感决策的工作流 | 客户服务中的退款审批 |
| **② 难以维护的规则** | 由于规则集庞大复杂而变得笨重，更新成本高或易出错的系统 | 供应商安全审查 |
| **③ 大量依赖非结构化数据** | 涉及解释自然语言、从文档中提取含义或与用户进行对话交互的场景 | 处理房屋保险索赔 |

> ⚠️ 在投入构建智能体之前，请验证你的用例是否能清晰满足这些标准。否则，确定性方案可能就足够了。

## 3. 智能体设计基础

在最基本的形式中，一个智能体由三个核心组件构成：

```
┌──────────────────────────────────────────────┐
│                  智能体 (Agent)                │
│                                              │
│  ① 模型 (Model)  ② 工具 (Tools)  ③ 指令 (Instructions)  │
│  ┌──────────┐  ┌──────────────┐  ┌───────────┐  │
│  │ LLM 驱动  │  │ 外部函数/API  │  │ 行为准则   │  │
│  │ 推理与决策 │  │ 用于执行行动  │  │ 与护栏定义  │  │
│  └──────────┘  └──────────────┘  └───────────┘  │
└──────────────────────────────────────────────┘
```

在 OpenAI 的 [Agents SDK](https://github.com/openai/openai-agents-python) 中，这看起来像这样——你也可以用你偏好的库或从头开始实现同样的概念：

```python
weather_agent = Agent(
    name="Weather agent",
    instructions="你是一个可以帮助用户了解天气的智能体。",
    tools=[get_weather],
)
```

### 3.1 选择模型

不同模型在任务复杂度、延迟和成本方面有不同的优势和权衡。正如我们将在下一节"编排"中看到的，你可能需要考虑在工作流的不同任务中使用不同的模型。

- 并非每项任务都需要最聪明的模型——一个简单的检索或意图分类任务可以用更小、更快的模型处理
- 而更困难的任务（如决定是否批准退款）可能受益于更强大的模型

**推荐方法**：

1. 先用最强大的模型为每项任务构建智能体原型，建立性能基线
2. 然后尝试换用较小的模型，看看能否仍达到可接受的结果
3. 这样你既不会过早限制智能体的能力，也能诊断出较小模型在哪些方面成功或失败

**选择模型的原则**：

```
① 建立评估体系 (Evals) → 确定性能基线
② 用最好的模型首先满足准确率目标
③ 用较小模型替换较大模型 → 优化成本和延迟
```

### 3.2 定义工具

工具通过调用底层应用或系统的 API 来扩展智能体的能力。对于没有 API 的遗留系统，智能体可以依赖"计算机使用"模型，通过网络和应用 UI 直接与这些系统交互——就像人类一样。

每个工具应有**标准化的定义**，实现工具与智能体之间灵活的多对多关系。文档完善、经过彻底测试且可重用的工具可以提高可发现性、简化版本管理并防止重复定义。

**智能体需要三种类型的工具：**

| 类型 | 描述 | 示例 |
|:---|:---|:---|
| **数据工具 (Data)** | 使智能体能够检索执行工作流所需的上下文和信息 | 查询交易数据库或 CRM 系统、读取 PDF 文档、搜索网络 |
| **行动工具 (Action)** | 使智能体能够与系统交互以执行操作 | 发送电子邮件和短信、更新 CRM 记录、将客服工单转给人工 |
| **编排工具 (Orchestration)** | 智能体本身可以作为其他智能体的工具——见编排中的"管理者模式" | 退款智能体、研究智能体、写作智能体 |

**使用 Agents SDK 定义工具的示例：**

```python
from agents import Agent, WebSearchTool, function_tool

@function_tool
def save_results(output):
    db.insert({"output": output, "timestamp": datetime.time()})
    return "File saved"

search_agent = Agent(
    name="Search agent",
    instructions="帮助用户搜索互联网，并在被要求时保存结果。",
    tools=[WebSearchTool(), save_results],
)
```

> 💡 随着所需工具数量的增加，考虑将任务拆分为多个智能体（见下一节"编排"）。

### 3.3 配置指令

高质量的指令对于任何 LLM 驱动的应用都至关重要，但对智能体来说尤为关键。清晰的指令能减少歧义，改善智能体的决策，从而实现更顺畅的工作流执行和更少的错误。

**智能体指令的最佳实践：**

| 最佳实践 | 说明 |
|---------|------|
| **📄 利用现有文档** | 使用现有的操作流程、支持脚本或政策文档来创建 LLM 友好的例程。在客服场景中，例程可以大致映射到知识库中的各个文章 |
| **🧩 提示智能体分解任务** | 从密集的资源中提供更小、更清晰的步骤，有助于减少歧义，帮助模型更好地遵循指令 |
| **🎯 定义清晰的操作** | 确保例程中的每一步都对应一个特定的操作或输出。明确说明操作（甚至用户消息的措辞），减少解释错误的空间 |
| **⚠️ 覆盖边界情况** | 现实世界的交互经常产生决策点。一个健壮的例程能预见常见的变化，并包含如何处理这些变化的指令 |

**自动生成指令的示例提示词：**

```
你是一位为LLM智能体编写指令的专家。将以下帮助中心文档转换为
一组清晰的指令，以编号列表的形式编写。该文档将成为LLM遵循的
政策。确保没有歧义，指令以对智能体的指示形式编写。

待转换的帮助中心文档是：{{help_center_doc}}
```

## 4. 编排（Orchestration）

有了基础组件，你可以考虑编排模式，使智能体能够有效地执行工作流。

虽然立即构建具有复杂架构的完全自主智能体很诱人，但客户通常通过**渐进式方法**取得更大的成功。

一般来说，编排模式分为两类：

| 模式 | 描述 | 适用场景 |
|:---|:---|:---|
| **① 单智能体系统** | 一个模型配备适当的工具和指令，在循环中执行工作流 | 大多数任务从这开始 |
| **② 多智能体系统** | 工作流执行分布在多个协调的智能体之间 | 单智能体无法应对复杂指令或工具过载时 |

---

### 4.1 单智能体系统

一个智能体可以通过**增量添加工具**来处理许多任务，保持复杂度的可控性，简化评估和维护。每个新工具扩展了它的能力，而不必过早地强制你编排多个智能体。

```
输入 → [智能体: 指令 + 工具 + 护栏 + 钩子] → 输出
```

**运行循环（Run Loop）**：每个编排方法都需要"运行"的概念，通常实现为一个循环，使智能体能够持续运行直到达到退出条件。常见的退出条件包括：

- 工具调用完成
- 产生特定的结构化输出
- 发生错误
- 达到最大轮数

在 Agents SDK 中：

```python
Agents.run(agent, [UserMessage("美国的首都是哪里？")])
```

> 💡 这种 **while 循环** 的概念是智能体运作的核心。

**有效策略：使用提示词模板**

与其为不同用例维护大量独立的提示词，不如使用一个灵活的**基础提示词模板**，接受策略变量：

```
你是一名呼叫中心智能体。你正在与 {{user_first_name}} 交互，
他已成为会员 {{user_tenure}} 时间。用户最常见的投诉是
{{user_complaint_categories}}。问候用户，感谢他们的忠诚，
并回答用户可能有的任何问题！
```

这种方法可以轻松适应各种情况，显著简化维护和评估。

---

### 4.2 何时考虑创建多个智能体

**我们的总体建议：首先最大化单个智能体的能力。** 更多智能体可以提供直观的概念分离，但也会引入额外的复杂性和开销，所以通常一个带工具的智能体就足够了。

**需要拆分智能体的两个信号：**

| 信号 | 说明 |
|:---|:---|
| **① 逻辑过于复杂** | 提示词包含大量条件语句，模板难以扩展时，考虑将每个逻辑段划分到独立的智能体中 |
| **② 工具过载** | 问题不只是工具的数量，而是**工具的相似性或重叠度**。如果明确定义工具名称、参数和描述后性能仍未改善，就使用多个智能体 |

---

### 4.3 多智能体系统

多智能体系统可以针对特定工作流和需求以多种方式设计，但我们的客户经验凸显了两种广泛适用的类别：

| 模式 | 结构 | 特点 |
|:---|:---|:---|
| **管理者模式 (Manager)** | 中央"管理者"通过工具调用协调多个专业智能体 | 一个智能体控制执行流程并接触用户 |
| **去中心化模式 (Decentralized)** | 多个智能体作为对等节点运行，根据各自的专长交接任务 | 智能体直接交接控制权 |

---

#### 🅰 管理者模式（Manager Pattern）

中央"管理者"LLM 通过工具调用无缝编排专业智能体网络。管理者智能地将任务委派给正确的智能体，轻松地将结果综合成连贯的交互。

```
用户: "把'你好'翻译成西班牙语、法语和意大利语！"
        │
        ▼
   ┌──────────┐
   │  管理者   │
   └────┬─────┘
     ┌──┼──┐
     ▼  ▼  ▼
  西语  法语 意语
  智能体 智能体 智能体
```

**Agents SDK 实现：**

```python
from agents import Agent, Runner

manager_agent = Agent(
    name="manager_agent",
    instructions=(
        "你是一个翻译智能体。你使用给你的工具进行翻译。"
        "如果要求多种翻译，你调用相关工具。"
    ),
    tools=[
        spanish_agent.as_tool(
            tool_name="translate_to_spanish",
            tool_description="将用户的消息翻译成西班牙语",
        ),
        french_agent.as_tool(
            tool_name="translate_to_french",
            tool_description="将用户的消息翻译成法语",
        ),
        italian_agent.as_tool(
            tool_name="translate_to_italian",
            tool_description="将用户的消息翻译成意大利语",
        ),
    ],
)
```

---

#### 🅱 去中心化模式（Decentralized Pattern）

智能体可以将工作流执行"交接"给彼此。交接是一种单向转移，允许一个智能体委派给另一个智能体。

```
用户: "我的订单在哪里？"
        │
        ▼
   ┌──────────┐
   │  分流智能体 │
   └────┬─────┘
    ┌───┼───┐
    ▼   ▼   ▼
  售后  销售  订单管理
                    ▼
              "正在运送中！"
```

**Agents SDK 实现：**

```python
from agents import Agent, Runner

technical_support_agent = Agent(
    name="技术支持智能体",
    instructions="你提供解决技术问题、系统中断或产品故障排除的专业协助。",
    tools=[search_knowledge_base]
)

sales_assistant_agent = Agent(
    name="销售助理智能体",
    instructions="你帮助企业客户浏览产品目录、推荐合适的解决方案并促进采购交易。",
    tools=[initiate_purchase_order]
)

order_management_agent = Agent(
    name="订单管理智能体",
    instructions="你协助客户查询订单跟踪、配送进度以及处理退货或退款。",
    tools=[track_order_status, initiate_refund_process]
)

triage_agent = Agent(
    name="分流智能体",
    instructions="你充当第一联系点，评估客户查询并将其及时引导至正确的专业智能体。",
    handoffs=[technical_support_agent, sales_assistant_agent,
              order_management_agent],
)
```

**两种模式的对比：**

```
管理者模式                         去中心化模式
────────                         ─────────
中央智能体控制流程                  智能体对等交接
智能体作为"工具"出现               智能体直接互相调用
适合需要统一用户体验的场景           适合专业智能体完全接管任务的场景
```

## 5. 护栏（Guardrails）

设计良好的护栏有助于你管理数据隐私风险（如防止系统提示泄露）或声誉风险（如确保品牌一致的模型行为）。你可以设置护栏来应对已识别的风险，并在发现新漏洞时增加更多防护。

> **把护栏想成一种分层防御机制。** 单一的护栏不太可能提供足够的保护，而使用多个专门的护栏共同作用，可以创建更有韧性的智能体。

```
用户输入 → LLM护栏 → 安全分类器 → Moderation API → 规则检查 → 通过？
                                                                    ↓
                                                             拒绝 / 继续
```

---

### 5.1 护栏的类型

| 类型 | 作用 | 示例 |
|:---|:---|:---|
| **🔍 相关性分类器** | 确保智能体回复不偏离预期范围 | 标记"帝国大厦有多高？"这样的与主题无关的问题 |
| **🛡️ 安全分类器** | 检测不安全输入（越狱、提示注入） | 检测试图提取系统提示的伪装请求 |
| **🔐 PII 过滤器** | 防止个人身份信息不必要地暴露 | 审查模型输出中的潜在 PII |
| **⚖️ Moderation API** | 标记有害或不适当的输入 | 仇恨言论、骚扰、暴力内容 |
| **🛠️ 工具安全措施** | 按风险等级评估每个工具 | 高风险操作触发暂停和人工升级 |
| **📋 基于规则的保护** | 简单的确定性措施 | 黑名单、输入长度限制、正则过滤 |
| **✅ 输出验证** | 确保回复符合品牌价值 | 通过提示工程和内容检查 |

**工具风险评级指南：**

| 风险等级 | 特征 | 处理方式 |
|:---|:---|:---|
| **低** | 只读访问、不可逆操作 | 允许自动执行 |
| **中** | 写访问但可逆 | 执行护栏检查后允许 |
| **高** | 不可逆、财务影响、敏感权限 | 需要人工确认 |

---

### 5.2 构建护栏

**有效的启发式方法：**

1. 🥇 **从数据隐私和内容安全开始**
2. 🥈 **基于真实世界的边界情况和失败逐步添加新护栏**
3. 🥉 **在安全性和用户体验之间寻求平衡，随着智能体的演进不断调整**

**Agents SDK 中设置护栏的示例：**

```python
@input_guardrail
async def churn_detection_tripwire(
    ctx, agent, input
) -> GuardrailFunctionOutput:
    result = await Runner.run(churn_detection_agent, input)
    return GuardrailFunctionOutput(
        output_info=result.final_output,
        tripwire_triggered=result.final_output.is_churn_risk,
    )

customer_support_agent = Agent(
    name="客户支持智能体",
    instructions="你是一名客户支持智能体，帮助客户解答问题。",
    input_guardrails=[
        Guardrail(guardrail_function=churn_detection_tripwire),
    ],
)
```

**关于人工干预：**

> 人工干预是一项关键的保障措施，使你能在不影响用户体验的情况下改善智能体的实际性能。在部署初期尤其重要，有助于发现失败、揭示边界情况并建立健壮的评估循环。

两个主要触发人工干预的情况：

| 触发条件 | 说明 |
|:---|:---|
| **⚠️ 超过失败阈值** | 设置智能体重试或操作的限制。如果超过限制，升级到人工干预 |
| **🔴 高风险操作** | 敏感、不可逆或高风险的行动，在信心足够前应触发人工监督。例如取消订单、授权大额退款、付款 |

## 6. 结论

智能体标志着工作流自动化的新时代——系统可以推理模糊性、跨工具采取行动、以高度自主性处理多步骤任务。

**构建可靠智能体的关键：**

```
① 坚实的基础
   └── 能力匹配的模型 + 定义清晰的工具 + 结构化的指令

② 匹配复杂度的编排
   └── 从单智能体开始 → 必要时演进到多智能体

③ 每个阶段都有护栏
   └── 输入过滤 → 工具使用 → 人工参与回路
```

**通往成功部署的路径不是全有或全无。从小处着手，用真实用户验证，逐步发展能力。** 有了正确的基础和迭代的方法，智能体可以提供真正的商业价值——不仅自动化任务，而是以智能和适应性的方式自动化整个工作流。
