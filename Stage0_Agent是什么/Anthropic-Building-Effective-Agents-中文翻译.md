## 1. 什么是智能体？

关于"智能体"的定义有很多，但在本文中，我们将"智能体"定义为：**大语言模型能够自主地决定过程、使用工具并循环执行任务，同时保持对如何完成任务的自主控制权的系统。**

然而，实际应用中的智能体落在一条光谱上——从简单的确定性工作流到完全自主的系统不等。

### 工作流（Workflow） vs 智能体（Agent）

| 维度 | 工作流（Workflow） | 智能体（Agent） |
|------|-------------------|----------------|
| **控制方式** | 预定义的代码路径 | LLM 动态控制流程 |
| **灵活性** | 低——固定步骤 | 高——自主决策 |
| **可预测性** | 高——输出可控 | 低——路径不确定 |
| **适用场景** | 确定性的、可预见的任务 | 开放式的、需要灵活应变的任务 |
| **典型实现** | 通过代码编排 LLM 调用 | LLM 自主决定工具调用和流程 |

> **关键区分**：核心差异在于**控制权在谁手里**——是人类开发者还是 LLM。工作流由代码驱动，智能体由模型驱动。

## 2. 何时（不）使用智能体

### 从最简单开始

我们最强烈的建议是：**从最简单的方案开始，只在有明确需要时才增加复杂度。**

这意味着：

1. **先不引入智能体** —— 仅用 LLM API 的单个调用是否能完成任务？
2. **再尝试简单工作流** —— 如果单次调用不够，能否用一个固定流程来解决？
3. **最后才考虑智能体** —— 只有问题确实需要 LLM 自主决策时，才引入完整的智能体架构。

### ✅ 适合使用智能体的场景

| 场景 | 原因 |
|------|------|
| **开放性问题** | 无法提前预知所有步骤，如开放式研究 |
| **需要大量工具调用** | 模型需要自主决定何时调用什么工具 |
| **需要从失败中恢复** | 智能体可以根据错误反馈调整策略 |
| **用户目标高度可变** | 每个用户的需求差异巨大，无法用固定流程覆盖 |

### ❌ 不适合使用智能体的场景

| 场景 | 更好的方案 |
|------|-----------|
| **固定流程的任务** | 使用工作流（Workflow） |
| **需要 100% 确定性的输出** | 纯代码实现 |
| **延迟敏感的场景** | 单次 LLM 调用或轻量级工作流 |
| **可以提前枚举所有路径的任务** | 条件逻辑 + 工作流 |

### 核心原则

> **简单 > 复杂**。最好的智能体系统不是功能最强大的，而是最简洁、最可维护的。

**决策树：**

```
你的任务需要...
│
├── 单次 LLM 调用就够？ → 直接用 API
│
├── 需要固定步骤编排？ → 用 Workflow
│   ├── 步骤固定、线性 → Prompt Chaining
│   ├── 需要分类分流 → Routing
│   ├── 可并行处理 → Parallelization
│   └── 需要多角色协作 → Orchestrator-Workers
│
└── 需要 LLM 自主决策？ → 用 Agent
    ├── 步骤不确定、需自主规划
    ├── 需要动态工具调用
    └── 需要从错误中恢复
```

## 3. 框架的选择

关于是否使用框架，我们的观点是：**框架可以加速原型开发，但不应该成为构建的起点。**

### 常见框架

| 框架 | 特点 | 适用场景 |
|------|------|---------|
| **LangChain / LangGraph** | 生态最丰富，有向图编排 | 需要复杂状态管理的场景 |
| **AutoGen** | 多 Agent 对话式协作 | 多角色协作的对话系统 |
| **CrewAI** | 角色化多 Agent 系统 | 任务可拆分的团队协作 |
| **直接调用 API** | 最轻量，完全可控 | 简单到中等复杂度的任务 |

### 我们的建议

1. **先用原生 API 构建** —— 直接调用 Claude API 或用简单的 Python 脚本搭建原型
2. **最小化抽象层** —— 每多一层抽象，就多一层调试成本
3. **仅在需要时引入框架** —— 当原生方案确实无法满足需求时，再评估框架
4. **理解框架底层** —— 不要盲目使用框架，要理解它到底帮你做了什么

> 💡 **经验之谈**：最成功的团队往往用最少的抽象。很多顶尖实现仅仅是用简单的 Python 循环 + LLM API 调用 + 工具函数就完成了。

## 4. 构建模块：工作流模式

我们在实践中观察到以下五种常见的工作流模式。它们可以作为构建智能体系统的基本组件。

### 4.1 提示链（Prompt Chaining）

**核心思想**：将一个任务分解为一系列顺序执行的 LLM 调用，每个调用的输出作为下一个调用的输入。

```
输入 → LLM 调用 A → 输出 A → LLM 调用 B → 输出 B → LLM 调用 C → 最终输出
```

**适用场景**：
- 任务可以清晰地分解为固定子步骤
- 每个步骤的输出需要后续处理
- 不需要步骤间的来回交互

**真实示例**：

```
步骤1: 生成文章大纲
步骤2: 根据大纲写每一节
步骤3: 翻译成目标语言
步骤4: 校对和润色
```

**代码示例**：

```python
def prompt_chain(input_text):
    # 步骤 1：生成大纲
    outline = llm_call(f"为以下主题生成文章大纲：{input_text}")
    
    # 步骤 2：撰写内容
    draft = llm_call(f"根据以下大纲撰写文章：{outline}")
    
    # 步骤 3：润色
    final = llm_call(f"润色以下文章，使其更加流畅专业：{draft}")
    
    return final
```

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 简单直观，易于实现 | 延迟累积（串行执行） |
| 每步结果可独立检查和调试 | 前一步的错误会向后传播 |
| 适合固定流程的任务 | 不够灵活，不适应变化 |

---

### 4.2 路由（Routing）

**核心思想**：先对输入进行分类，然后根据分类结果将其路由到对应的专用处理器。

```
            ┌─────────┐
输入 ──────→│ 分类器   │
            └────┬────┘
         ┌───────┼──────────┐
         ▼       ▼          ▼
     ┌──────┐ ┌──────┐ ┌────────┐
     │处理A  │ │处理B  │ │ 处理C   │
     └──────┘ └──────┘ └────────┘
         │       │          │
         └───────┴──────────┘
                  ▼
              最终输出
```

**适用场景**：
- 输入类型明显可分，不同类别需要不同的处理逻辑
- 每个类别的处理可以独立优化
- 需要为不同查询分配不同成本和延迟

**真实示例**：

```python
def route_input(user_query):
    # 分类
    category = classify_query(user_query)
    
    if category == "客服售后":
        return handle_after_sales(user_query)
    elif category == "产品咨询":
        return handle_product_inquiry(user_query)
    elif category == "技术支持":
        return handle_tech_support(user_query)
    elif category == "简单问答":
        return simple_qa(user_query)
    else:
        return fallback_handler(user_query)
```

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 每个处理模块可以高度专业化 | 分类准确率是关键瓶颈 |
| 可以分离关注点，降低复杂度 | 需要为每个类别维护专用逻辑 |
| 延迟更低（只需走一条路径） | 类别边界模糊时可能误分 |

---

### 4.3 并行化（Parallelization）

**两个主要变体**：

#### 🅰 分段模式（Sectioning）

将任务拆分为可并行处理的独立子任务：

```
输入
├── LLM A：分析技术可行性
├── LLM B：分析市场需求
├── LLM C：分析竞争格局
└── LLM D：分析风险因素
         │
     ┌───┴───┐
     │ 汇总合并 │
     └───────┘
         ▼
     综合报告
```

**适用场景**：任务的各个子部分可以独立处理，互不依赖。

#### 🅱 投票模式（Voting）

用多个 LLM 调用以不同角度或提示处理同一任务：

```
输入
├── LLM A（乐观视角）→ 方案 A
├── LLM B（保守视角）→ 方案 B
├── LLM C（创新视角）→ 方案 C
└── LLM D（批判视角）→ 方案 D
         │
     ┌───┴───┐
     │ 投票/评估 │
     └───────┘
         ▼
     最佳方案
```

**适用场景**：
- 需要对同一问题进行多角度评估
- 需要提高决策的稳健性
- 可以通过投票机制减少偏差

**代码示例**：

```python
def parallel_voting(question):
    # 并行的多角度分析
    responses = []
    
    def ask_with_perspective(perspective):
        return llm_call(f"从{perspective}的角度分析：{question}")
    
    # 并行调用（实际可用线程池/异步）
    perspectives = ["技术可行", "商业价值", "风险评估", "用户体验"]
    results = [ask_with_perspective(p) for p in perspectives]
    
    # 汇总投票
    final = llm_call(f"综合以下多个角度的分析，给出最终结论：\n" + "\n".join(results))
    return final
```

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 速度更快（并行执行） | Token 消耗增加（多路调用） |
| 结果更稳健（多角度验证） | 需要处理结果一致性 |
| 适合探索性、开放性分析 | 实现相对复杂 |

### 4.4 编排器-工作者（Orchestrator-Workers）

**核心思想**：一个中央编排器（Orchestrator）LLM 负责分解任务并分派给多个工作者（Worker）LLM，最后汇总结果。

```
                  ┌──────────────┐
                  │  编排器 LLM   │
                  │  (Orchestrator)│
                  └──────┬───────┘
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │ 工作者 A  │   │ 工作者 B  │   │ 工作者 C  │
    │ (Worker)  │   │ (Worker)  │   │ (Worker)  │
    └──────────┘   └──────────┘   └──────────┘
          │              │              │
          └──────────────┼──────────────┘
                         ▼
                  ┌──────────────┐
                  │  结果汇总     │
                  └──────────────┘
```

**适用场景**：
- 复杂任务无法提前固定子任务划分
- 子任务的数量和类型在运行时才能确定
- 需要动态规划和灵活分配

**真实示例**：代码修改任务

```python
class OrchestratorWorker:
    def run(self, task):
        # 编排器：规划需要修改的文件
        plan = self.orchestrator.plan(task)
        # {'files': ['auth.py', 'db.py'], 'changes': [...]}
        
        # 工作者：并行修改各个文件
        results = []
        for file_change in plan['files']:
            worker = Worker(file_change)
            result = worker.modify()
            results.append(result)
        
        # 编排器：检查结果一致性
        final_check = self.orchestrator.verify(results)
        return final_check
```

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 高度灵活，可适应复杂任务 | 编排器是单点瓶颈 |
| 动态规划优于固定流程 | 编排器决策质量至关重要 |
| 适合需要分层理解的任务 | 实现复杂度较高 |

### 4.5 评估器-优化器（Evaluator-Optimizer）

**核心思想**：两个 LLM 协作——一个负责生成（Generator），另一个负责评估（Evaluator），通过循环迭代不断优化输出。

```
          ┌──────────────┐
     ┌───→│  生成器 LLM   │
     │    │  (Generator)  │
     │    └──────┬───────┘
     │           │ 生成
     │           ▼
     │    ┌──────────────┐
     │    │  评估器 LLM   │
     │    │ (Evaluator)   │
     │    └──────┬───────┘
     │           │
     │    ┌──────┴──────┐
     │    通过？        不通过？
     └────返回继续优化───┘
     │
     ▼
  最终输出
```

**适用场景**：
- 输出质量可以通过明确标准评估
- 迭代改进能显著提升效果
- 需要高质量、高精度的输出

**真实示例**：翻译优化

```python
def evaluator_optimizer(text, source_lang, target_lang):
    max_iterations = 3
    translation = llm_call(f"将以下{source_lang}翻译成{target_lang}：{text}")
    
    for i in range(max_iterations):
        # 评估
        feedback = llm_call(f"""评估以下翻译质量：
        原文：{text}
        译文：{translation}
        请指出问题和改进建议：""")
        
        # 如果评估满意，直接返回
        if "无问题" in feedback or "完美" in feedback:
            return translation
        
        # 根据反馈优化
        translation = llm_call(f"""根据以下反馈改进翻译：
        原文：{text}
        当前译文：{translation}
        反馈：{feedback}
        改进后的译文：""")
    
    return translation
```

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 输出质量高，迭代优化 | Token 消耗大（多轮迭代） |
| 有明确的改进机制 | 延迟较高 |
| 适合高标准的内容生成 | 评估标准定义有难度 |

## 5. 智能体（Agents）

当工作流模式不足以应对任务的复杂性和不确定性时，智能体（Agent）就派上用场了。

### 智能体的核心循环

```
接收目标 → 感知环境 → 思考规划 → 执行行动 → 观察结果 → 评估调整 → 继续循环
```

智能体与工作流最大的不同在于：**智能体自己决定下一步做什么**。

### 常见智能体实现方式

#### 🅰 基于代码的智能体（Code-Defined Agent）

系统提示词 + 工具描述的 Agent，由 LLM 自主决策：

```python
def agent_loop(task, tools):
    messages = [{"role": "user", "content": task}]
    
    while True:
        response = llm_call(messages, tools)
        
        if response.has_tool_call():
            # 智能体决定调用工具
            tool_result = execute_tool(response.tool_call)
            messages.append({"role": "assistant", "content": response})
            messages.append({"role": "tool", "content": tool_result})
        else:
            # 智能体决定给出最终答案
            return response.content
```

#### 🅱 基于模型的智能体（Model-Defined Agent）

使用具有内置 Agent 能力的模型（如 Claude 的 Tool Use 功能），让模型原生支持工具调用。

### 工具设计的三个关键原则

1. **给工具足够的上下文** —— 每个工具的描述应该像给新入职的工程师写文档一样清晰
2. **工具接口要一致** —— 同类工具的输入输出格式应保持一致
3. **考虑组合使用** —— 工具应设计为可组合的，而不是大而全的单体工具

### 应用技巧

| 技巧 | 说明 |
|------|------|
| **让规划透明化** | 要求 Agent 展示思考过程，便于调试和纠偏 |
| **设置合理的限制** | 如最大迭代次数、超时时间 |
| **提供错误恢复机制** | Agent 调用工具失败后应能自动重试或换方案 |
| **记录完整轨迹** | 保存每一步的 Thought-Action-Observation 日志 |
| **人工介入点** | 在关键决策点允许人工确认 |

### 两种智能体的对比

| 维度 | 代码定义 Agent | 模型定义 Agent |
|------|---------------|---------------|
| **控制粒度** | 细——可以精确控制每一步 | 粗——交给模型自主决定 |
| **实现复杂度** | 更高——需要自己管理循环 | 更低——模型原生支持 |
| **灵活性** | 高——可以插入任意逻辑 | 受限于模型能力 |
| **适合场景** | 需要精细控制和定制逻辑 | 标准的工具调用场景 |

## 6. 总结

### 我们的核心建议

1. 🎯 **从简单开始** —— 能不引入智能体就不引入，能用工作流就不用 Agent
2. 🧩 **使用可组合的模式** —— 五种工作流模式就像乐高积木，灵活组合
3. 🔧 **精心设计工具** —— 工具是智能体的"手"，好的工具设计至关重要
4. 📊 **保持可见性** —— 让智能体的思考过程和工具调用对开发者透明
5. 🏗️ **渐进式复杂化** —— 只在有明确需要时才增加复杂度

### 设计原则一览

| 原则 | 描述 |
|------|------|
| **KISS** | Keep It Simple, Stupid — 保持简单 |
| **渐进增强** | 从简单方案开始，逐步增加复杂度 |
| **可观察性** | 每个步骤都应该可追踪、可调试 |
| **容错设计** | 工具调用失败是常态，要有恢复机制 |
| **成本意识** | 每个额外的 LLM 调用都有成本，要精打细算 |

### 需要记住的金句

> 💬 **"最成功的智能体实现，不是功能最强大的，而是最简洁、最可维护的。"**

> 💬 **"框架擅长加速原型开发，但简单代码更擅长构建生产系统。"**

> 💬 **"你不需要一个智能体框架来构建智能体——你只需要一个 LLM API 和一个好的循环逻辑。"**

## 7. 附录 1：智能体实战

### 案例 1：客户支持智能体

**架构**：
```
用户消息
    │
    ▼
┌──────────────┐
│  路由分类器    │ → 简单问题 → 快速回复
└──────┬───────┘
       │ 复杂问题
       ▼
┌──────────────┐
│  客户支持Agent │
│              │
│  ┌──────────┐│
│  │ 工具库    ││
│  │ ├ 查订单  ││
│  │ ├ 退换货  ││
│  │ ├ 查政策  ││
│  │ └ 升级人工││
│  └──────────┘│
└──────────────┘
       │
       ▼
   回复用户
```

**实现要点**：
- Agent 根据对话历史自动判断需要哪些信息
- 在必要时升级到人工客服
- 每次工具调用都记录在案，便于审计

### 案例 2：编码智能体

**架构**：
```
用户需求
    │
    ▼
┌──────────────┐
│  Code Agent   │
│              │
│  1. 理解需求   │
│  2. 搜索代码库  │
│  3. 规划修改方案 │
│  4. 执行修改    │
│  5. 测试验证    │
│  6. 提交 PR    │
└──────────────┘
```

**实现要点**：
- 需要访问代码库的搜索工具
- 修改前先生成明确的计划
- 修改后自动运行测试
- 支持从编译错误中恢复

## 8. 附录 2：工具的提示工程

工具是智能体能力的扩展，工具的设计质量直接影响智能体的表现。

### 工具设计的 ACI（Agent-Computer Interface）原则

正如 UI 设计关注人机交互，ACI 关注的是 **智能体与工具之间的交互**。

#### 好的工具描述应该：

```python
# ❌ 差的工具描述
tools = [{
    "name": "search",
    "description": "搜索工具",
    "parameters": {...}
}]

# ✅ 好的工具描述
tools = [{
    "name": "web_search",
    "description": """执行网络搜索以获取实时信息。
当你需要回答涉及以下内容的问题时应使用此工具：
- 当前事件和新闻
- 你的训练数据中没有的信息
- 需要验证的事实数据
- 产品价格、规格等动态信息

输入：一个精确的搜索查询字符串，就像你在 Google 中输入的搜索词一样。
注意：查询越具体，结果越好。建议使用 2-5 个关键词。""",
    "parameters": {
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "搜索查询词。使用精确的关键词，例如 '2024 年诺贝尔物理学奖' 而不是 '最近有什么新闻'"
            }
        },
        "required": ["query"]
    }
}]
```

#### 工具设计的最佳实践

| 实践 | 说明 |
|------|------|
| **详细的描述** | 告诉模型什么时候用、怎么用、有什么注意事项 |
| **明确的参数说明** | 每个参数都应该有清晰的描述和格式要求 |
| **提供示例** | 在描述中包含使用示例 |
| **错误信息要清晰** | 工具返回的错误应包含可操作的修复建议 |
| **保持接口一致** | 类似功能的工具输入输出格式应统一 |

#### 常见问题

| 问题 | 解决方案 |
|------|---------|
| Agent 用错了工具 | 增强工具描述，明确区分不同工具的适用场景 |
| Agent 传错了参数 | 参数描述要包含格式、范围、单位等约束 |
| Agent 不会从错误中恢复 | 工具的错误返回应包含"如何修复"的建议 |
| Agent 过度调用工具 | 增加工具调用的成本提示，或在提示词中鼓励更谨慎地使用 |
